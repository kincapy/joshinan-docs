# システムコンセプト

## このシステムが目指す世界

**人間は「入力」だけをやれば、そこから先の作業・出力はすべてシステムが完結する。**

日本語学校には、事務スタッフや日本語教師の前に「学生」という人間がいる。
事務的な仕事の時間を削減し、人と向き合うための時間を作る。
このシステムはそのために存在する。

## 3つの設計思想

### 1. 人間の入力を最小化する

人間に期待するのは「入力」だけ。しかもその入力自体も極力簡素にする。

| 従来の入力 | このシステムでの入力 |
|---|---|
| 紙や在留カードを見て、フォームに手入力 | PDFや画像をアップロード → 自動でDB登録 |
| パソコンのフォームから面談記録を入力 | スマートフォンから音声入力 |
| 画面を開いてフィルターをかけてデータを探す | 「先月の未納者出して」とプロンプトで聞く |
| 住所変更のフォームを開いて修正 | 「この学生、A物件からB物件に引っ越したから直して」と伝える |

**理想: 人間にフォームを一切触らせない。プロンプトかPDFアップロードで全てが終わる。**

### 2. 人間の「考える作業」を最小化する

通常、業務上の判断には法律を読む、入管のサイトを調べる、同僚に聞くといった作業が発生する。
このシステムでは、業務ナレッジをシステム内に充足させることで、チャットボットとの対話で業務知識を取得できるようにする。

| 従来 | このシステム |
|---|---|
| 法律や告示基準を自分で調べる | チャットボットに聞く |
| 入管のサイトで手続きを確認する | チャットボットに聞く |
| 先輩や同僚に運用ルールを聞く | チャットボットに聞く |

ナレッジの源泉はこのVitePressのMarkdownファイル群。
マニュアルを更新すれば、ドキュメントサイトもAIの回答も同時に更新される。

### 3. 業務管理をシステムが主導する

人間が業務管理をすると、ヒューマンエラーで「本来やるべき報告ができていなかった」という事態が起きる。
このシステムでは、発生する業務をシステムが把握し、人間に対して業務を管理する。

**従来:** 人間がタスクを覚えておいて、自分で管理する
**このシステム:** システムがタスクを管理し、人間にアラートを出す

## 2種類の入力トリガー

システムに発生する入力は、大きく2種類に分けられる。

### イベントベース（不定期・突発的）

学校生活の中で発生する出来事がトリガーになる入力。

| トリガー | 入力の例 |
|---|---|
| 学生が入学した | 入学情報の登録 |
| 学生が退学した | 退学処理、入管への届出 |
| 学生が引っ越した | 住所変更、入管への届出 |
| 学生と面談した | 面談記録の登録（音声入力） |
| 学費が振り込まれた | 入金消込 |

→ これらは**人間が発生時に入力する**。ただし入力方法は最小化する。

### スケジュールベース（定期・予定済み）

年間スケジュールや法的義務から、あらかじめ発生時期がわかっている業務。

| 時期 | 業務の例 |
|---|---|
| 毎月 | 出席率の集計・報告 |
| 半年に1回 | 定期報告（入管） |
| 年度末 | 年度末報告 |
| 学生ごと | ビザ更新（在留期間満了の2〜3ヶ月前） |

→ これらは**システムが業務の発生を管理し、人間にアラートを出す**。
人間がやるのは、そのタスクを完了させること。

## スケジュールベース業務の具体例: ビザ更新

ビザ更新は、スケジュールベース業務の中でも特に複雑な例。

```
1. システムが在留期限を監視し、更新時期にアラートを出す
2. 学生ごとに、ビザ更新に必要な情報一覧を表示
3. 各情報のステータスを管理（未回収 / 回収済み）
   - 例: 給与明細 → 未回収（学生から受け取る必要あり）
   - 例: 出席率証明 → 回収済み（システムが自動生成）
4. 人間は未回収の情報を学生から受け取り、システムにアップロード
5. 全情報が「回収済み」になったら、システムがビザ更新資料を自動作成
6. 人間は出力された資料を確認して提出するだけ
```

**ポイント:** 人間がやるのは「学生から書類を受け取ってアップロードする」ことだけ。
資料の作成・情報の突合・進捗管理はすべてシステムが行う。

## 技術的な実現手段

| コンセプト | 実現手段 |
|---|---|
| 入力の最小化（PDF → DB） | OCR / AI による文書解析 |
| 入力の最小化（音声 → テキスト） | 音声認識API |
| 入力の最小化（プロンプト → DB操作） | Claude API Tool Use |
| 考える作業の最小化 | チャットボット + VitePressナレッジ |
| 業務管理のシステム主導 | スケジュールエンジン + アラート機能 |
| フォームレスな操作 | プロンプトベースUI |

## 今の段階でやること

このコンセプトを実現するための基盤として、今はVitePressのドキュメントを充実させる。

- Markdownファイル = チャットボットのナレッジソース
- 業務フロー = システムの処理フロー設計の土台
- データ定義 = DBテーブル設計・Tool Use関数の土台
- タスク一覧 = スケジュールエンジンの管理対象

**ドキュメントの質 = システムの質** になるため、ここの充実が最優先。
